# 申根签材料助手移动端紧凑布局设计

## 背景

当前 `src/components/visa-assistant.tsx` 及相关子组件已经具备完整的签证表单、行程卡、解释信、材料清单和导出能力，但视觉节奏更偏桌面端：

- 页面外层容器在手机端保留了较多上下留白。
- 首屏头部卡片、免责声明、主区块头部与正文都使用了偏宽松的 `padding`。
- `TripBasicsSection`、`DayCard`、预览卡等二级卡片在手机端继续沿用较大的段落间距。
- 结果是手机浏览时首屏有效信息偏少，用户需要更快滚动才能看到基础信息和第一张行程卡的主要内容。

用户已经确认希望采用“方案 B：按断点做中度紧凑化”，并认可了对应的独立预览稿方向。

## 目标

- 在不改变现有业务逻辑的前提下，降低移动端页面留白占比。
- 让用户在手机首屏更快看到“标题区 + 基础信息入口 + 第一张 Day 卡”的有效内容。
- 保持桌面端现有视觉节奏基本稳定，仅在移动端和小屏断点做明显收紧。
- 保持按钮、输入框、选择器等触控区域可用，不通过牺牲点击面积换取紧凑感。
- 让三个 tab（行程单、解释信、材料清单）在移动端都遵循统一的紧凑布局规则。

## 非目标

- 不改日期两步选择逻辑。
- 不改城市 / 景点 / 机场 / 行程生成 / 导出状态机等业务逻辑。
- 不重排表单字段顺序。
- 不引入折叠、分页、吸顶工具栏等新的交互结构。
- 不重做桌面端视觉方案。

## 采用方案

采用“按断点做中度紧凑化”的响应式样式方案。

核心原则：

- 优先压缩外层容器、卡片内边距、段落间距和标题层级。
- 保留现有组件结构与交互行为。
- 通过默认类收紧移动端布局，再用 `sm:` / `md:` / `lg:` 在较大屏幕恢复当前视觉密度。
- 优先处理用户最明显感受到留白的区域：页面外层、首屏头部、`TripBasicsSection`、`DayCard`、预览卡片。

## 设计细节

## 整体布局

### 页面外层容器

`src/components/visa-assistant.tsx` 中外层 `main` 当前使用偏保守的宽度和较大的上下留白。

移动端调整方向：

- 页面宽度略放宽，让内容更贴近屏幕边界。
- 上下 `padding` 从当前的宽松节奏收紧到中等节奏。
- 各个一级区块之间的 `margin-bottom` 与 `space-y` 统一降低一档。

目标效果：

- 减少手机首屏上下空带。
- 不让内容贴边过度，仍保留必要安全边距。

### 首屏头部

首屏头部卡片继续保留：

- 主标题
- 辅助说明
- tab 切换按钮
- 重置按钮

移动端调整方向：

- 头部卡片 `padding` 从 `p-6` 收到 `p-4` 级别。
- 主标题从当前移动端 `text-3xl` 降一档到 `text-2xl` 级别。
- 副文案行高轻微收紧。
- 按钮组保留横向换行，但按钮外观调整为更接近紧凑 chip 的节奏，减少换行后的总高度。

目标效果：

- 头部仍有明确识别度，但不再占据过多首屏空间。

### 免责声明

免责声明继续作为独立提示存在，不与头部合并，也不删除。

移动端调整方向：

- 缩小提示卡的 `padding`。
- 保留圆角和边框，但整体高度更薄。
- 文案字号与行高略收紧。

目标效果：

- 保留风险提示的可见性。
- 避免在头部之后形成第二段大面积留白。

### 一级主卡片

`签证行程单生成`、`签证解释信生成`、`签证申请材料准备清单` 三个主卡片统一采用同一组移动端紧凑规则。

移动端调整方向：

- 区块头部由当前 `px-6 py-5` 收到 `px-4 py-4` 级别。
- 区块正文由当前 `px-6 py-6` 收到 `px-4 py-4` 级别。
- 区块正文内的 `space-y-5`、`gap-4/5` 统一收紧。

桌面端策略：

- 在 `md` 或更高断点恢复当前或接近当前的宽松节奏。

## 行程单 Tab

### 基础信息卡

`src/components/visa-form/trip-basics-section.tsx` 作为进入行程单的第一块内容，是本次紧凑化的重点区域之一。

移动端调整方向：

- `CardHeader` 与 `CardContent` 的上下 `padding` 和 `gap` 统一收紧。
- 字段块之间的 `space-y-1.5` 继续保留，但整体网格间距减少。
- “出发日期 / 返回日期”保留双列展示，以减少纵向长度。
- 日期选择按钮和静态日期展示块只缩小外层间距和容器 padding，不改变两步选取逻辑。
- “生成行程安排表”按钮保持足够点击面积，仅收紧与上下内容的距离。

目标效果：

- 基础信息区更像连续录入表单，而不是松散堆叠的模块。

### DayCard

`src/components/visa-form/day-card.tsx` 是移动端滚动长度的主要来源，因此需要系统性收紧，但不改变现有信息结构。

移动端调整方向：

- `CardHeader` 的顶部和底部留白降低。
- `DAY X` 标签、标题、副说明继续保留，但副说明与头部块的垂直间距减少。
- `CardContent` 的 `space-y-5 pt-5` 收到 `space-y-3/4` 和更小的顶部 `padding`。
- 内部每个 section 的 `p-4` 降到移动端 `p-3` 级别。
- 城市子卡 `p-4` 收到 `p-3`，景点标签区、提示文案区和删除按钮区域的间距同步收紧。
- 住宿信息、交通方式、飞机附加字段、时间参考提示条统一减少 `gap`、`margin-top` 和容器 padding。

明确不变部分：

- 城市与景点的按行绑定模型不变。
- 删除城市、勾选景点、切换交通方式、选择机场、填写航班号等行为不变。

目标效果：

- 每张 Day 卡在手机端滚动长度明显缩短。
- 用户可以在一屏中看到更多“真实内容”而不是大块留白。

### 生成与预览区域

行程单 Tab 中的生成按钮、中文预览、英文预览也需要同步收紧，以保持整体节奏一致。

移动端调整方向：

- 生成按钮区域减少上下留白。
- 预览外层网格保持单列到双列的现有响应式结构，不改变信息顺序。
- `GeneratedPreview` 的头部与正文 `padding` 收紧。
- 可编辑预览区保留最小高度和编辑能力，只减少卡片外层的空带。
- PDF 导出进度条状态块保持状态机与文案不变，仅缩小外层 `padding` 和内部间距。

## 解释信 Tab

`src/components/visa-assistant.tsx` 中解释信区域也遵循相同紧凑规则。

移动端调整方向：

- 标题区 `margin-bottom` 与 `space-y` 收紧。
- 顶部双列字段在小屏下继续换回单列或紧凑双列，不改字段逻辑。
- 保证选项容器、文本域、生成按钮、手写签名提示条之间的垂直间距减少。
- 中英文编辑区保留现有双栏响应式切换，只缩小外层卡片内边距与编辑块周边间距。

目标效果：

- 解释信 tab 的视觉密度和行程单 tab 保持一致。

## 材料清单 Tab

材料清单 tab 继续保留逐条勾选模式。

移动端调整方向：

- 标题区与正文区的 `padding` 收紧。
- 每条清单卡片继续保留边框、badge 和说明文案，但 `p-3`、行高、元素间距统一降低一档。
- checkbox 与文案对齐方式保持不变，避免点击体验变差。

目标效果：

- 手机端一屏能看到更多清单项。

## 组件级实施边界

本次样式调整优先集中在以下文件：

- `src/components/visa-assistant.tsx`
- `src/components/visa-form/trip-basics-section.tsx`
- `src/components/visa-form/day-card.tsx`
- `src/components/visa-form/generated-preview.tsx`

如有必要，可在不改变语义的前提下对少量按钮、卡片或通用类名做同步调整，但不应扩散为整站风格重构。

## 错误处理与兼容性

本次为纯样式密度调整，不新增新的异步流程或新的状态。

需要注意的风险：

- 过度压缩可能导致按钮换行后反而更高，因此移动端按钮缩小应以“减少外围空白”为主，而不是极端缩短内容本身。
- 过度压缩可能影响长文案可读性，因此副文案、免责声明、提示条只做中度收紧。
- `DayCard` 内部 section 较多，需确保移动端收紧后仍能明确区分日期、城市、住宿和交通几类内容。

## 验证要点

### 手工验证

- 手机宽度下首屏能更快看到基础信息区和第一张 Day 卡。
- 头部标题、副文案、tab 按钮和免责声明的总高度明显下降。
- `TripBasicsSection` 中字段仍清晰可点，日期选择交互不受影响。
- `DayCard` 中城市、景点、住宿、交通区块仍有清晰层次，不会因为收紧而显得拥挤难读。
- 中文 / 英文预览区仍可正常编辑，导出按钮可正常点击。
- 解释信和材料清单两个 tab 在手机端同样呈现统一的紧凑节奏。
- 桌面端视觉不出现明显回归。

### 工程验证

本次以以下方式为主：

- TypeScript / IDE diagnostics
- 本地页面手工检查移动端视口
- 必要时运行项目现有校验命令

## 验收标准

- 用户主观感受到“手机浏览时页面留白明显减少”。
- 视觉变化集中在移动端，不影响当前桌面端主要布局节奏。
- 没有业务逻辑回归，没有交互行为变化。
- 三个 tab 的移动端紧凑化规则保持一致，不出现某个 tab 明显更松或更挤的情况。
