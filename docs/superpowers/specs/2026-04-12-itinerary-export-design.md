# 中英文行程单 DOCX/PDF 导出设计

## 背景

当前签证行程单能力已经具备：

- 左侧维护基础信息、每日路线、住宿与交通信息。
- 右侧生成中英文行程单预览。
- 预览内容由 `src/components/visa-assistant.tsx` 直接拼接 HTML 字符串，再交给 `GeneratedPreview` 渲染。

当前实现可以满足浏览器内预览，但还不能满足新的导出需求：

- 需要导出中文版行程单与英文版行程单。
- 每种语言都要支持 `DOCX` 与 `PDF` 两种格式。
- 导出内容必须与页面预览保持一致，不能出现“预览一套、导出另一套”的漂移。

## 目标

- 支持中英文行程单导出为真正的 `DOCX` 文件。
- 支持中英文行程单导出为 `PDF` 文件。
- 让预览、DOCX、PDF 三条链路共享同一份文档语义模型。
- 保持现有行程单业务规则不变，包括日期、路线、景点、住宿、交通、跨城交通、机场与办理入住逻辑。
- 在现有 UI 基础上增加清晰可用的导出入口。

## 非目标

- 不包含解释信的 DOCX/PDF 导出。
- 不改签证材料清单模块。
- 不引入服务端文档生成能力。
- 不做批量 zip 打包下载。
- 不重构整个行程编辑表单，只聚焦行程单预览与导出链路。

## 采用方案

采用“统一文档模型 + 多渲染器输出”的方案。

核心思想：

- 先把当前 `visa-assistant.tsx` 中直接拼接 HTML 的逻辑抽成结构化文档模型。
- 页面预览改为由“文档模型 -> HTML 渲染器”输出。
- `DOCX` 导出由“文档模型 -> DOCX 导出器”输出。
- `PDF` 导出由“文档模型 -> 打印模板页面 -> 浏览器打印”输出。

这样可以让预览与导出都依赖同一份业务数据，避免后续维护时内容不一致。

## 方案对比

### 方案 A：继续以 HTML 为中心扩展导出

- 保留当前 HTML 字符串拼接。
- `DOCX` 从 HTML 或临时转换结构生成。
- `PDF` 直接基于预览 DOM 打印。

优点：

- 初期改动较小。

缺点：

- HTML 既承担业务拼装又承担展示，职责混杂。
- `DOCX` 很难可靠地从当前 HTML 逆向出稳定结构。
- 后续新增字段时，预览和导出容易漂移。

### 方案 B：统一文档模型后分别导出

- 抽取文档模型，统一表达标题、头信息、表头、表格行。
- 预览、DOCX、PDF 均从该模型生成。

优点：

- 结构清晰，长期维护成本低。
- 更适合继续扩展解释信导出等能力。
- 更容易做文件命名、语言区分和导出校验。

缺点：

- 需要先做一层重构，再接入导出。

### 方案 C：直接用前端 PDF 库硬排版

- `DOCX` 用 `docx` 库。
- `PDF` 用 `jspdf` 或 `pdf-lib` 直接在前端绘制表格。

优点：

- 下载体验统一，不依赖浏览器打印。

缺点：

- 中文、表格、自动换页、长文本与样式控制复杂。
- 在当前项目里实现成本高，调试成本也高。

## 选择理由

本次选择方案 B。

原因：

- 当前预览逻辑集中在 `src/components/visa-assistant.tsx`，已经具备稳定的业务拼装能力，但输出层过于耦合在 HTML 字符串上。
- 导出需求一旦落地，继续把 HTML 当作唯一中间格式会让 DOCX 和 PDF 实现变得脆弱。
- 统一模型后，可以用最小额外改动复用既有业务逻辑，又能显著降低后续演进成本。

## 文档模型设计

### 模型职责

文档模型只表达“行程单是什么”，不表达“如何渲染成哪种格式”。

模型中需要包含：

- 文档标题
- 副标题或申请人头信息
- 语言标识
- 列定义
- 每日行程行
- 文件名建议

### 建议结构

建议新增一个共享类型，例如：

```ts
type ItineraryDocument = {
  locale: "zh" | "en";
  title: string;
  headerFields: Array<{ label: string; value: string }>;
  columns: Array<{ key: string; label: string }>;
  rows: Array<Record<string, string>>;
  fileName: string;
};
```

说明：

- `headerFields` 用于承载申请人、护照号、出发地等信息。
- `columns` 用于表达表头顺序。
- `rows` 用于表达每天的行程内容。
- `fileName` 在导出时直接复用，避免在导出器内再次拼装命名逻辑。

### 构建方式

新增一个 builder，例如：

- `buildItineraryDocument(days, formState, locale)`

它复用现有逻辑中的：

- 路线计算
- 景点中英文转换
- 酒店信息拼接
- 交通方式拼接
- 跨城交通补充
- 飞机起降信息
- 办理入住补充

这样可以最大限度保持现有文案和业务规则不变。

## 预览链路调整

### 当前问题

当前 `generateItinerary()` 直接在页面组件中调用 `setZhItinerary()` / `setEnItinerary()` 写入整段 HTML。

这会导致：

- 预览格式与业务规则耦合。
- 导出无法安全复用。
- 页面组件职责过重。

### 调整方式

将预览改为两步：

- 先构建 `ItineraryDocument`
- 再通过 `renderItineraryHtml(document)` 生成 HTML

这样：

- `GeneratedPreview` 仍然可以保持接受 `html` 字符串的接口。
- 页面结构变化最小。
- 预览与导出共享同一份输入。

## DOCX 导出设计

### 导出方式

使用前端 `docx` 库生成真正的 `.docx` 文件。

导出内容包括：

- 文档标题
- 申请人头信息
- 行程单表格

表格列顺序：

- 中文：`天数`、`日期（星期）`、`城市`、`景点`、`住宿`、`交通方式`
- 英文：`Day`、`Date (Weekday)`、`City`、`Attractions`、`Accommodation`、`Transportation`

### 实现边界

- 单次只导出一种语言。
- 每次点击导出按钮直接触发下载。
- 不追求 Word 中百分之百像素级还原浏览器预览，只要求信息完整、结构清晰、表格可读。

### 文件命名

建议：

- 中文：`schengen-itinerary-zh-YYYY-MM-DD.docx`
- 英文：`schengen-itinerary-en-YYYY-MM-DD.docx`

## PDF 导出设计

### 导出方式

采用“打印模板页面 + 浏览器打印”的方式导出 PDF。

流程：

- 基于 `ItineraryDocument` 渲染一份独立打印模板。
- 打开新窗口或新标签页。
- 注入专用打印样式，控制为 A4 纵向布局。
- 自动触发打印，由用户保存为 PDF。

### 交互流程细化

PDF 导出不是瞬时下载，而是一个多阶段流程，因此需要显式的进度反馈。

建议将交互拆为四个阶段：

- `准备文档`：根据当前语言构建 `ItineraryDocument`。
- `生成打印页`：创建打印窗口并写入 HTML 与打印样式。
- `打开打印对话框`：等待新窗口完成渲染后触发 `print()`。
- `等待用户完成`：等待用户在浏览器打印对话框中保存 PDF 或取消。

### 进度条设计

主页面在点击 `导出中文 PDF` 或 `导出英文 PDF` 后，显示一个轻量级进度条提示区。

建议使用固定的阶段式进度，而不是伪造持续增长动画：

- `25%`：文档模型已构建完成
- `50%`：打印页已创建完成
- `75%`：打印对话框已触发
- `100%`：导出流程已结束，打印窗口已完成回调或关闭

说明：

- `100%` 表示“本次导出交互流程结束”，不强行宣称用户一定已经保存成功。
- 浏览器原生打印对话框无法可靠感知“用户确实点击了保存”，因此最终状态只能区分为“流程完成”或“流程中断/失败”。

### 状态文案

建议为 PDF 导出维护一组明确的阶段文案：

- `正在准备 PDF 内容...`
- `正在生成打印页...`
- `正在打开打印对话框...`
- `请在系统打印窗口中保存为 PDF`
- `PDF 导出流程已结束`
- `PDF 导出已取消`
- `PDF 导出失败，请检查浏览器弹窗权限后重试`

### 状态机

建议维护一个独立的 PDF 导出状态机，例如：

```ts
type PdfExportStatus =
  | { phase: "idle"; progress: 0 }
  | { phase: "building"; progress: 25; message: string }
  | { phase: "rendering"; progress: 50; message: string }
  | { phase: "printing"; progress: 75; message: string }
  | { phase: "waiting"; progress: 75; message: string }
  | { phase: "done"; progress: 100; message: string }
  | { phase: "cancelled"; progress: 100; message: string }
  | { phase: "error"; progress: 100; message: string };
```

状态流转建议：

- `idle -> building`
- `building -> rendering`
- `rendering -> printing`
- `printing -> waiting`
- `waiting -> done | cancelled | error`
- 任一阶段异常可直接进入 `error`

### 完成态与取消态判定

浏览器打印能力存在平台差异，因此完成态判定应尽量保守：

- 如果打印窗口成功打开，且触发了打印，再收到 `afterprint` 回调，可进入 `done`。
- 如果用户关闭打印窗口，或主页面收到窗口关闭信号但未明确失败，可进入 `cancelled`。
- 如果新窗口创建失败、被拦截、写入异常或打印触发异常，则进入 `error`。

说明：

- `done` 表示浏览器打印链路顺利走完，不等价于“文件一定已保存到磁盘”。
- `cancelled` 表示用户主动中断或关闭流程，属于可预期行为，不应提示为错误。

### 主页面 UI 形态

建议在导出按钮附近展示一个短时出现的状态条，包含：

- 当前导出目标，如 `中文 PDF` 或 `英文 PDF`
- 一条线性进度条
- 当前阶段文案
- 在 `waiting` 阶段补充辅助说明：`请在弹出的系统窗口中选择“另存为 PDF”`

建议行为：

- 状态条仅在 PDF 导出过程中显示。
- `done` / `cancelled` / `error` 后保留 2 到 3 秒，再自动收起。
- 导出进行中禁用两个 PDF 按钮，防止并发打开多个打印窗口。
- DOCX 按钮可保持可用，或统一禁用；默认推荐统一禁用，避免用户在打印进行中重复触发多种导出。

### 选择理由

- 对表格、长文本和中文内容更稳定。
- 避免直接使用前端 PDF 绘图库做复杂排版。
- 与当前 HTML 预览链路天然接近，但依然保持模型分层。

### 样式要求

- 页面尺寸为 A4。
- 标题、头信息和表格有明确间距。
- 表格列头固定清晰。
- 单元格允许自动换行。
- 打印页背景尽量简化，避免无关装饰。

### 文件命名

建议：

- 中文：`schengen-itinerary-zh-YYYY-MM-DD.pdf`
- 英文：`schengen-itinerary-en-YYYY-MM-DD.pdf`

## UI 设计

### 导出入口位置

导出按钮放在“签证行程单生成”区域头部，靠近当前 `ItineraryToolbar`。

建议保留现有：

- `新增一天`
- `生成行程单`

并补充四个导出按钮：

- `导出中文 DOCX`
- `导出英文 DOCX`
- `导出中文 PDF`
- `导出英文 PDF`

### 按钮状态

- 在尚未生成行程单时，四个导出按钮应为禁用状态。
- 鼠标悬停或辅助文案提示：`请先生成行程单`。
- 生成内容后再允许导出。

### 交互反馈

- 点击 DOCX 导出后，若成功则直接开始下载。
- 点击 PDF 导出后，主页面先显示阶段式进度条，再由用户在浏览器打印窗口中完成保存。
- 若浏览器拦截新窗口，应进入错误态，并提示允许弹窗或打印窗口。
- 若用户关闭打印流程，则展示取消态提示，而不是统一报错。

## 模块拆分建议

建议新增以下职责模块：

- `itinerary-document.ts`
  - 负责类型定义与文档模型构建
- `itinerary-html.ts`
  - 负责把文档模型渲染成 HTML 预览
- `itinerary-docx.ts`
  - 负责 DOCX 导出
- `itinerary-print.ts`
  - 负责打印页 HTML 与 PDF 导出

这样可以避免继续把所有字符串拼装逻辑堆积在 `visa-assistant.tsx` 中。

## 错误处理

### 未生成内容

- 用户未点击“生成行程单”前尝试导出，应阻止操作并提示先生成。

### DOCX 生成失败

- 捕获生成或下载异常。
- 提示 `DOCX 导出失败，请稍后重试。`
- 不清空页面已有预览。

### PDF 打印失败

- 若新窗口无法打开或打印页构建失败，提示用户检查浏览器弹窗权限。
- 不影响原页面编辑状态。
- 若用户主动取消打印，则提示 `PDF 导出已取消`，不记为错误。

### 字段缺失

延续当前页面已有的兜底逻辑：

- 中文使用 `未填写`
- 英文使用 `N/A`
- 其他表格单元格使用 `-`

## 边界处理

- 如果当天没有景点，景点列仍显示兜底值。
- 如果当天交通包含飞机但机场或航班号不完整，交通列仍输出已填写部分。
- 如果当天与前一天跨城，跨城交通与办理入住逻辑保持现状，不因导出链路变化而改变。
- 如果用户编辑预览区内容但未重新生成，导出以“最近一次生成的文档模型”为准，不尝试从可编辑 DOM 反向解析内容。

说明：

- 这是有意为之，避免把 `contentEditable` DOM 当作业务数据源。
- 如果后续需要“编辑后导出”，应单独设计受控编辑模型，本次不纳入范围。

## 验证要点

### 手工验证

- 生成行程单后，可看到中文与英文预览仍正常显示。
- 点击 `导出中文 DOCX` 后，能下载并在 Word 中打开。
- 点击 `导出英文 DOCX` 后，能下载并在 Word 中打开。
- 点击 `导出中文 PDF` 后，可进入打印页并保存 PDF。
- 点击 `导出英文 PDF` 后，可进入打印页并保存 PDF。
- 中文与英文的标题、头信息、表头和表格内容一致且语言正确。
- 跨城交通、飞机信息、办理入住等现有规则在导出文件中仍生效。

### 自动化验证

当前仓库尚未配置成熟的文档导出测试基础设施，本次以：

- `pnpm typecheck`
- IDE diagnostics
- 浏览器内手工下载验证

作为主要验证手段。

## 风险

- `docx` 库接入后，表格宽度和长文本换行需要适配，避免文档在 Word 中可读性差。
- 浏览器打印在不同环境下可能存在细微差异，因此打印样式需要尽量收敛。
- 当前预览是 `contentEditable`，如果用户手动修改预览内容，导出结果可能与可见 DOM 不一致；本次设计明确以文档模型为准。
- `visa-assistant.tsx` 已承担较多业务逻辑，本次实施中如果不及时抽离 builder/render/export 模块，代码复杂度会继续上升。

## 验收标准

- 页面支持导出中文行程单 `DOCX`。
- 页面支持导出英文行程单 `DOCX`。
- 页面支持导出中文行程单 `PDF`。
- 页面支持导出英文行程单 `PDF`。
- 预览、DOCX、PDF 共用同一份文档模型。
- 导出后的内容与当前生成规则一致，不丢失路线、景点、住宿、交通、跨城交通与飞机信息。
- 未生成行程单时，导出按钮不可用。
