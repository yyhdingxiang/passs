# 行程安排卡片按城市逐行绑定设计

## 背景

当前 `src/components/visa-form/day-card.tsx` 已从旧版“结构展示优先”的布局改为更偏主表单的布局，但仍存在两个核心问题：

- `游览城市` 与 `游览景点` 仍然是独立录入区域，用户无法直观看出某个景点属于哪个城市。
- `+新增游览城市` 不是直接新增一行“城市 + 该城市景点”的输入结构，交互仍偏间接。

同时，用户补充了新的业务规则：

- 如果当天主城市和前一天主城市不同，行程单中除了当日交通信息，还要补充前一日到当日完整跨城交通。
- 如果当天交通方式为飞机，机场候选需要覆盖前一日城市机场与当日城市机场。
- 删除“已录入城市与景点”模块，不再保留第二层结构管理区。

## 目标

- 让每一行只表达一个游览城市及其对应景点。
- `游览景点` 必须从属于该行 `游览城市`，不能作为独立全局输入区存在。
- 点击 `+新增游览城市` 时，直接追加一行新的“城市单选 + 景点多选”输入。
- 保留当前整体视觉风格，不改整体卡片风格体系。
- 补全行程单跨城交通与飞机机场候选逻辑。

## 非目标

- 不重构整个 `DayItem` 数据模型。
- 不改基础信息区 `trip-basics-section.tsx`。
- 不引入新的表单库或新的日期/级联组件库。
- 不改右侧预览整体布局。

## 采用方案

采用“按城市逐行绑定”的单层表单方案。

核心思想：

- 一行对应一个城市段。
- 这一行左侧负责选择一个 `国家/城市`。
- 这一行右侧负责给该城市选择多个景点。
- 增加新城市时，直接新增新行，而不是弹出或跳转到别的管理区。

## 交互设计

### 每行结构

每一行城市段包含：

- `游览城市`：单选，使用 `国家/城市` 级联选择器。
- `游览景点`：多选，且仅作用于当前这一行对应的城市。
- `删除城市`：仅删除当前这一行城市及其景点。

视觉上：

- 一行内横向排布“城市选择”和“景点选择”。
- 在窄屏下允许纵向换行。
- 每行保持当前卡片体系中的圆角、边框、背景、字号风格。

### 新增城市

- 初始显示至少一行城市段。
- 用户点击 `+新增游览城市` 后：
  - 在当前最后一行下方新增一行空白城市段。
  - 已填写行保持不变。
  - 不新增任何摘要标签，不插入额外展示模块。

### 景点选择规则

- 只有当该行已选城市后，才允许为这一行补充景点。
- 这一行的景点展示与删除只作用于当前行。
- 不再保留独立的全局“游览景点”区域。

### 删除“已录入城市与景点”模块

- 删除当前底部的“已录入城市与景点”模块。
- 删除国家段、城市、景点的维护能力，直接内嵌到每一行城市段中完成。

## 数据结构映射

### 继续复用现有结构

继续复用：

- `DayItem`
- `CountryRow`
- `CityRow`

不新增新的顶层业务模型。

### UI 到数据的映射方式

现有结构中：

- `CountryRow` 下有多个 `cityRows`

新的 UI 语义中：

- 每一个可见输入行都对应一个 `cityRow`
- 该行所属国家来自其所在的 `CountryRow.country`
- 该行景点来自该 `cityRow.scenics`

也就是说，UI 不再围绕“国家段汇总展示”组织，而是围绕“城市行”组织。

### 新增城市的数据写入

点击 `+新增游览城市` 时：

- 如果已有可复用的空位，则复用空位。
- 否则追加新的空城市行。
- 新行初始无城市、无景点。

当用户在某一行选择 `国家/城市` 后：

- 更新该行所属国家
- 更新该行城市名
- 保留其他行不变

当用户在某一行选择景点后：

- 仅更新该行的 `scenics`

## 行程单生成规则

### 办理入住

保留已确认规则：

- 如果当天主城市与前一天主城市不同，则在“景点”列追加 `办理入住`

英文版对应：

- `Hotel check-in`

### 跨城交通补充

新增规则：

- 如果当天主城市与前一天主城市不同：
  - 当天交通列除了原有当天交通信息
  - 还要补充 `前一日主城市 -> 当日主城市` 的跨城交通描述

交通方式取值：

- 使用当天填写的交通方式

说明：

- 用户已明确确认跨城移动视为“当天发生”，因此使用当天交通信息。

### 飞机机场候选

如果当天交通方式包含 `飞机`：

- 起降机场候选应覆盖：
  - 前一日主城市机场
  - 当日所有已选城市机场

这样可支持：

- 前一日城市飞到当日主城市
- 当天多个城市中选择实际落地/起飞机场

## UI 变更点

### `src/components/visa-form/day-card.tsx`

需要调整为：

- 删除当前独立的城市摘要标签 `div`
- 删除独立的全局景点录入区
- 删除“已录入城市与景点”模块
- 改成逐行渲染城市段
- 保留 `日期 / 住宿信息 / 交通方式` 的现有主表单排布思路

每行城市段需支持：

- 选择城市
- 选择该城市景点
- 删除该城市行

### `src/components/visa-assistant.tsx`

需要调整：

- 计算主城市时继续使用第一有效城市
- 生成行程单时补跨城交通说明
- 计算机场候选时，将前一日主城市机场合并进当天候选

## 边界处理

- 新增的空白城市行未选择城市时，不应影响其他已填写行。
- 如果某一行还未选择城市，则该行景点选择应为空或禁用态。
- 如果删除某一行城市，其他行内容和顺序保持稳定。
- 如果当天没有前一日主城市，则不补跨城交通。
- 如果跨城但缺少机场或航班号，则交通列仍展示已填部分，不额外报错。

## 验证要点

### 手工验证

- 点击 `+新增游览城市` 后，页面下方直接新增一整行输入。
- 第一行已选城市和景点不因新增第二行而变化。
- 每一行景点只属于该行城市。
- 删除某一行城市后，仅该行消失。
- “已录入城市与景点”模块不再出现。
- 当天与前一天主城市不同，行程单交通列包含跨城交通说明。
- 当天交通为飞机时，可选到前一日城市机场和当日城市机场。

当前仓库未配置现成组件测试基础设施，本次以：

- `pnpm typecheck`

作为主要验证手段。

## 风险

- 现有 `CountryRow -> cityRows` 是按“国家段”建模，UI 改为“逐行城市”后，需要谨慎处理同一国家下多个城市的映射与删除。
- 如果城市行编辑逻辑写得不清晰，可能出现改动一行影响其他行的回写问题。
- 飞机场候选合并前一日城市后，需要避免重复机场项。

## 验收标准

- 一行只展示一个游览城市和其对应景点。
- 每新增一个游览城市，就新增一整行输入。
- 景点不再作为独立全局区域存在，而是绑定到对应城市行。
- 页面中不再出现城市摘要标签 `div`。
- 页面中不再出现“已录入城市与景点”模块。
- 跨城时，行程单交通列补充前一日到当日的跨城交通。
- 飞机机场候选支持前一日城市机场与当日城市机场。
