# 出行日期区间选择设计

## 背景

当前“出行日期”字段位于 `src/components/visa-form/trip-basics-section.tsx`，使用 `Popover + Calendar` 组合实现日期选择，底层为 `react-day-picker` 的区间模式。

现状已经具备区间选择基础能力，但交互目标需要明确为“用户一次连续选择一个出行周期”，而不是让用户理解为逐天离散点击或分别维护两个单独日期字段。

## 目标

- 用户通过一次区间选择完成整个出行周期录入。
- 保持现有数据结构不变，继续输出 `tripStartDate` 与 `tripEndDate` 两个字符串字段。
- 保持“生成行程安排表”逻辑不变，继续依赖起止日期自动生成天数。
- 在尽量小的改动范围内提升区间选择的可理解性与可操作性。

## 非目标

- 不引入新的日期组件库。
- 不改造成拖拽划选交互。
- 不改成两个独立的可编辑日期输入框。
- 不调整父组件 `visa-assistant.tsx` 的业务数据结构。

## 方案对比

### 方案 A：保留当前区间日历并补齐交互

做法：
- 继续使用当前 `Calendar` 的 `mode="range"`。
- 明确交互为“先选开始日期，再选结束日期”。
- 选完整段后关闭弹层。
- 增加“清空日期”入口，并优化未完成选择时的提示文案。

优点：
- 与当前实现最一致。
- 改动最小，风险最低。
- 不影响现有父层状态与生成逻辑。

缺点：
- 仍依赖用户理解区间选择规则，需要通过文案和状态展示降低误解。

### 方案 B：改为开始/结束两个字段

做法：
- 拆分为两个独立日期选择入口。
- 由组件内部或父层合成区间。

优点：
- 语义直观。

缺点：
- 表单更重。
- 与当前“一个字段表达一个周期”的使用场景不完全匹配。
- 需要额外处理双向同步和边界状态。

### 方案 C：支持拖拽连续划选

做法：
- 增强日历交互，支持鼠标拖拽或滑动形成区间。

优点：
- 交互感更强。

缺点：
- 不属于当前组件默认能力。
- 实现与兼容性成本明显更高。
- 对当前需求属于过度设计。

## 采用方案

采用方案 A。

原因：
- 已满足“连续选择一个周期”的核心需求。
- 对现有 UI、状态结构和后续行程生成逻辑影响最小。
- 更适合在当前代码基础上快速稳定落地。

## 交互设计

### 主交互

- 用户点击“出行日期”按钮后打开日历弹层。
- 第一次点击某一天时，设置开始日期。
- 第二次点击某一天时，设置结束日期，并自动形成包含中间所有日期的完整区间。
- 当开始日期和结束日期都已选定后，自动关闭弹层。

### 已选区间后的行为

- 当用户已经完成一次区间选择后，再点击任意某一天，应当开始新一轮区间选择。
- 新一轮选择开始时：
  - 该日期成为新的开始日期。
  - 结束日期清空，等待用户选择新的结束日期。

### 展示文案

- 未选择任何日期时：`选择出发和返回日期`
- 仅选择开始日期时：`YYYY-MM-DD - 请选择结束日期`
- 已完成区间选择时：`YYYY-MM-DD - YYYY-MM-DD`

### 辅助操作

- 在日期选择区域提供“清空日期”入口。
- 清空后：
  - `tripStartDate` 置空
  - `tripEndDate` 置空
  - 按钮文案恢复到未选择状态

## 数据流设计

### 现有数据结构

保持以下接口不变：

- `tripStartDate: string`
- `tripEndDate: string`
- `onTripRangeChange(range: { start: string; end: string }): void`

### 组件内状态

继续保留：

- `datePickerOpen`
- `selectedRange`

其中 `selectedRange` 仍由 `tripStartDate` 和 `tripEndDate` 派生，不新增额外的业务状态。

### 区间更新规则

`handleRangeSelect` 继续作为唯一出口，负责把 `DateRange` 转换为字符串：

- `from` 有值时写入 `start`
- `to` 有值时写入 `end`
- 无值时写入空字符串

该函数还负责在 `from` 与 `to` 同时存在时关闭弹层。

## UI 变更点

### `src/components/visa-form/trip-basics-section.tsx`

预期修改：

- 保留现有 `Popover`、触发按钮和 `Calendar`。
- 继续使用 `mode="range"` 和双月显示。
- 增加“清空日期”按钮或等价轻量入口。
- 根据 `selectedRange` 保持当前按钮文本和下方只读回显同步。

### `src/components/ui/calendar.tsx`

默认不计划修改。

仅当验证发现当前封装影响区间高亮、中间态展示或点击后的重选行为时，再做最小修正。当前静态检查下不预期需要改动。

## 错误与边界处理

- 当用户只选了开始日期但关闭弹层时，保留开始日期，结束日期为空。
- 当父层传入的日期字符串为空或非法时，组件按未选择处理。
- 当用户重新开始一次选择时，不保留上一次的结束日期。
- 不在当前需求内新增禁用日期、最小日期、最大日期等限制。

## 测试与验证

### 手工验证

- 打开日期弹层后，点击开始日期，再点击结束日期，确认完整区间被选中。
- 完成区间选择后，确认弹层自动关闭。
- 已有区间时再次点击任意日期，确认开始一轮新选择。
- 清空日期后，确认起止日期与展示文案均恢复初始状态。
- 点击“生成行程安排表”后，确认仍按区间天数生成日期列表。

### 自动化测试

本次优先手工验证。

原因：
- 改动集中在交互层。
- 当前仓库未显式展示该组件附近已有日期交互测试模式。
- 若实际实现中出现复杂条件分支，再补充针对 `TripBasicsSection` 的 focused test。

## 实施范围

- 必改：`src/components/visa-form/trip-basics-section.tsx`
- 可能涉及：`src/components/ui/calendar.tsx`
- 不改：`src/components/visa-assistant.tsx` 的状态结构与 `autoBuildDays` 逻辑

## 风险

- 如果当前工作树中 `trip-basics-section.tsx` 已有未提交修改，实施时需要在最新文件内容基础上增量合并，避免覆盖现有改动。
- 如果 `react-day-picker` 在当前版本的 range 行为与预期有差异，可能需要对“重选起点”逻辑做轻量封装处理。

## 验收标准

- 用户可以通过“开始日期 + 结束日期”完成一次连续区间选择。
- 选择完成后，按钮和下方字段正确展示开始与结束日期。
- 用户可重新开始一个新区间，不需要先手动逐项修改起止日期。
- 用户可清空已选日期。
- 行程生成功能保持现有行为不变。
